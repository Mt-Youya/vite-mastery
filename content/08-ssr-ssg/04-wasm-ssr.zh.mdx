## 为什么要在 SSR 里用 Wasm

WebAssembly 在浏览器里的典型用途:

- 图片/视频处理(ffmpeg.wasm)
- 加密算法
- 物理模拟、游戏引擎

**SSR 里的 Wasm**用途:

- 服务端图片处理(代替 sharp)
- 高性能 Markdown 解析
- 服务端加密(比 Node.js 原生实现更快)
- Rust 编写的 SQL 解析器、数据处理

## Vite 8 的 Wasm SSR 支持

在 client 端,Vite 一直支持 Wasm:

```ts
import init, { encode } from "./codec.wasm"
await init()
const result = encode("hello")
```

在 SSR 环境,Vite 8 扩展了对 Node.js 的 Wasm 支持:

```ts
// entry-server.tsx
import init, { processImage } from "./image-processor.wasm"

// Vite 8:在 SSR 环境里也能正常初始化和调用
await init()

export async function render(url: string) {
  // 在服务端调用 Wasm 函数
  const processedBuffer = await processImage(rawImageBuffer)
  // ...
}
```

## 配置 Wasm SSR

```ts filename="vite.config.ts"
export default defineConfig({
  ssr: {
    // 不要把 .wasm 文件 external
    noExternal: [/\.wasm$/],
  },
  optimizeDeps: {
    // 在预构建阶段处理 wasm
    include: ["my-wasm-package"],
  },
})
```

## 实际使用示例:Markdown 解析

```ts filename="entry-server.tsx"
// 使用 Rust 写的 Markdown 解析器(比 marked 快 10x)
import init, { renderMarkdown } from "./markdown-parser.wasm"

let wasmInitialized = false

async function ensureWasmInitialized() {
  if (!wasmInitialized) {
    await init()
    wasmInitialized = true
  }
}

export async function render(url: string) {
  await ensureWasmInitialized()

  const markdownContent = await fetchContent(url)
  // 在 Node.js SSR 里调用 Wasm 渲染 Markdown
  const htmlContent = renderMarkdown(markdownContent)

  return renderToString(<Article html={htmlContent} />)
}
```

## Wasm SSR vs Client Wasm

| 维度       | Client Wasm      | SSR Wasm       |
| ---------- | ---------------- | -------------- |
| 运行环境   | 浏览器           | Node.js        |
| 初始化时机 | 页面加载时       | 服务器启动时   |
| 内存共享   | 每个用户一份     | 所有请求共享   |
| 网络开销   | 用户需下载 .wasm | 服务器本地加载 |
| 适合场景   | 交互式处理       | 静态内容生成   |

**服务端初始化一次,全部请求共用**——这是 SSR Wasm 的核心优势。

## 局限与注意事项

1. **不是所有 Wasm 都支持 SSR**:依赖 DOM / browser API 的 Wasm 不能在 Node.js 里运行
2. **内存管理**:长期运行的 SSR 服务器里,Wasm 内存泄漏是实际问题
3. **Node.js Wasm API**:Node.js 的 `WebAssembly` API 与浏览器基本兼容,但有细节差异

## 什么时候不要用 Wasm SSR

Wasm SSR 很诱人,但不是所有高性能场景都该上 Wasm:

| 场景 | 更好的选择 |
| --- | --- |
| Node 原生库成熟且有维护者 | 直接用原生库,例如图片处理优先评估 `sharp` |
| 逻辑很轻,瓶颈在数据库或网络 | 优化 I/O,不要引入 Wasm 初始化成本 |
| 模块依赖 DOM、Canvas、WebGL | 放到 client 或 worker,SSR 无法直接运行 |
| 每个请求都要创建新实例 | 先设计实例池或单例初始化,否则性能会更差 |

Wasm 适合把“纯计算、可复用、初始化后长期运行”的逻辑放到服务端。例如 Markdown、代码高亮、压缩、解析器、编解码器。它不适合替代所有 Node.js 包。

## 生产部署检查

上线前至少检查这些点:

1. `.wasm` 文件是否被打进服务端产物,部署平台能否读取。
2. 初始化是否只发生一次,而不是每个请求重复执行。
3. 并发请求是否共享同一个实例,共享时有没有可变全局状态。
4. 错误路径是否能降级,例如 Wasm 初始化失败时返回 500 还是切到 JS 实现。
5. 监控里是否能看到 Wasm 初始化耗时、内存增长和请求耗时。

如果部署在 serverless 或 edge runtime,还要确认该平台允许加载 Wasm,以及冷启动时初始化成本是否可接受。
