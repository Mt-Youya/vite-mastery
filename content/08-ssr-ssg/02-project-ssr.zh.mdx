## 目标架构

我们要做的是低层 SSR，不是复刻 Next.js:

```text
浏览器请求 /
  → Node HTTP server
  → dev: Vite middleware 处理 /@vite/client、源码、HMR
  → 加载 server entry
  → render(url) 生成 HTML + 初始数据
  → transformIndexHtml 处理模板
  → 注入 <!--ssr-outlet--> 与 <!--ssr-data-->
  → 浏览器 hydrateRoot 接管交互
```

开发和生产最大的差异是“server entry 从哪里来”:

| 阶段 | server entry 来源 | 为什么 |
| ---- | ----------------- | ------ |
| dev | Vite 即时转换后的源码 | 支持 HMR、源码堆栈、按需转换 |
| prod | `dist/server/entry-server.js` | 不在生产环境依赖 dev server |

这也是 SSR 项目一般要构建两次的原因:client build 给浏览器，server build 给 Node/Worker runtime。

## 项目结构

```text
examples/ssr-with-env-api/

├── vite.config.ts
├── server.js
├── index.html
└── src/
    ├── App.tsx
    ├── main.tsx
    └── entry-server.tsx
```

三个入口分工:

| 文件 | 运行环境 | 职责 |
| ---- | -------- | ---- |
| `src/App.tsx` | client + server | 共享 React 树 |
| `src/entry-server.tsx` | server | 把 React 树渲染成 HTML |
| `src/main.tsx` | client | hydrate 服务端 HTML |

## Vite 配置

```ts filename="vite.config.ts"
import { defineConfig } from "vite"
import react from "@vitejs/plugin-react"

export default defineConfig({
  plugins: [react()],
  environments: {
    client: {},
    ssr: {
      resolve: {
        conditions: ["node", "import"],
        noExternal: ["react", "react-dom"],
      },
    },
  },
  build: {
    manifest: true,
  },
})
```

配置含义:

- `plugins: [react()]`:client 端 JSX、Fast Refresh、server 端 React transform 都需要它。
- `environments.ssr.resolve.conditions`:让 SSR 环境按 Node 条件解析包入口。
- `noExternal`:示例里把 React 相关包交给 Vite/Rolldown 处理，避免 ESM/CJS 边界在演示里干扰主线。
- `build.manifest`:生产 HTML 注入和 preload 优化会用到。

真实项目里不要机械照抄 `noExternal: ["react", "react-dom"]`。多数 Node SSR 项目会 externalize 大部分依赖，只把确实需要 Vite 转换的包放进 `noExternal`。

## 共享组件

```tsx filename="src/App.tsx"
interface AppProps {
  url?: string
  data?: { message: string }
}

export function App({ url = "/", data }: AppProps) {
  return (
    <main>
      <h1>Vite SSR 演示</h1>
      <p>当前路径: {url}</p>
      {data ? <p>服务端数据: {data.message}</p> : null}
      <button onClick={() => alert("client 侧交互正常!")}>测试 Hydration</button>
    </main>
  )
}
```

这个组件会执行两次:

1. 服务端执行一次，产出 HTML 字符串。
2. 浏览器执行一次，`hydrateRoot` 绑定事件并复用现有 DOM。

所以共享组件里不要直接访问 `window`、`document`、`localStorage`。如果必须访问，放进 `useEffect` 或用 `import.meta.env.SSR` 分支保护。

## 服务端入口

```tsx filename="src/entry-server.tsx"
import { renderToString } from "react-dom/server"
import { App } from "./App"

export interface RenderResult {
  html: string
  data: { message: string }
}

export async function render(url: string): Promise<RenderResult> {
  const data = {
    message: `在服务端获取的数据 (${new Date().toISOString()})`,
  }

  const html = renderToString(<App url={url} data={data} />)
  return { html, data }
}
```

`render()` 返回 `html` 和 `data`。`html` 放进模板的 `<!--ssr-outlet-->`，`data` 序列化后交给浏览器复用。

如果不注入初始数据，浏览器 hydrate 时会重新请求一次数据，很容易出现:

- 服务端 HTML 和客户端首帧不一致
- hydration warning
- 页面闪烁
- 同一接口被请求两次

## 客户端入口

```tsx filename="src/main.tsx"
import { hydrateRoot } from "react-dom/client"
import { App } from "./App"

declare global {
  interface Window {
    __SSR_DATA__?: { message: string }
  }
}

hydrateRoot(
  document.getElementById("root")!,
  <App url={window.location.pathname} data={window.__SSR_DATA__} />
)
```

这里用的是 `hydrateRoot`，不是 `createRoot`。区别:

| API | 适合场景 |
| --- | -------- |
| `createRoot` | 纯客户端渲染，容器里通常是空的 |
| `hydrateRoot` | SSR 后接管已有 HTML |

如果 SSR 页面用了 `createRoot`，React 会丢弃服务端 HTML 重新渲染，SSR 的首屏优势就没了。

## HTML 模板

```html filename="index.html"
<!doctype html>
<html lang="zh">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Vite SSR</title>
  </head>
  <body>
    <div id="root"><!--ssr-outlet--></div>
    <script>
      window.__SSR_DATA__ = <!--ssr-data-->;
    </script>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>
```

生产项目里序列化数据时要防 XSS。不要直接把用户输入拼进 `<script>`:

```ts
html.replace("<!--ssr-data-->", JSON.stringify(data))
```

更稳妥的实现需要转义 `<`、`</script>`、U+2028、U+2029 等字符，或使用成熟序列化库。这里为了聚焦 Vite SSR 主流程，示例保持最小。

## 抽象 server entry 加载器

为了从 `ssrLoadModule` 平滑迁移到 `ModuleRunner.import()`，不要把加载逻辑散落在请求处理里。先抽象:

```ts filename="server-entry-loader.ts"
interface ServerEntry {
  render(url: string): Promise<{ html: string; data: unknown }>
}

interface ServerEntryLoader {
  load(url: string): Promise<ServerEntry>
}
```

dev 兼容实现:

```ts filename="server-entry-loader.ts"
import type { ViteDevServer } from "vite"

export function createSsrLoadModuleLoader(vite: ViteDevServer): ServerEntryLoader {
  return {
    async load(_url) {
      return vite.ssrLoadModule("/src/entry-server.tsx") as Promise<ServerEntry>
    },
  }
}
```

未来迁移到 ModuleRunner 时，目标形态是把内部替换成:

```ts filename="server-entry-loader.ts"
export function createModuleRunnerLoader(runner: { import<T>(url: string): Promise<T> }): ServerEntryLoader {
  return {
    async load(_url) {
      return runner.import<ServerEntry>("/src/entry-server.tsx")
    },
  }
}
```

关键不是这段接口多复杂，而是**请求处理层不关心入口模块怎么被执行**。这样官方 API 变化时，只替换 loader。

## HTTP server

```js filename="server.js"
import { createServer as createHttpServer } from "node:http"
import { readFileSync } from "node:fs"
import { createServer as createViteServer } from "vite"

const isProd = process.argv.includes("--prod")

async function startServer() {
  let vite
  let template

  if (!isProd) {
    vite = await createViteServer({
      server: { middlewareMode: true },
      appType: "custom",
    })
  } else {
    template = readFileSync("dist/client/index.html", "utf-8")
  }

  const server = createHttpServer(async (req, res) => {
    const url = req.url ?? "/"

    if (!isProd && vite) {
      await new Promise((resolve) => {
        vite.middlewares(req, res, resolve)
      })
      if (res.writableEnded) return
    }

    try {
      let appHtml
      let data
      let html

      if (!isProd && vite) {
        const { render } = await vite.ssrLoadModule("/src/entry-server.tsx")
        ;({ html: appHtml, data } = await render(url))

        const rawTemplate = readFileSync("index.html", "utf-8")
        html = await vite.transformIndexHtml(url, rawTemplate)
      } else {
        const { render } = await import("./dist/server/entry-server.js")
        ;({ html: appHtml, data } = await render(url))
        html = template
      }

      html = html
        .replace("<!--ssr-outlet-->", appHtml)
        .replace("<!--ssr-data-->", JSON.stringify(data))

      res.setHeader("Content-Type", "text/html; charset=utf-8")
      res.end(html)
    } catch (error) {
      if (vite) vite.ssrFixStacktrace(error)
      console.error(error)
      res.statusCode = 500
      res.end(String(error))
    }
  })

  server.listen(3000, () => {
    console.log("SSR server: http://localhost:3000")
  })
}

startServer()
```

dev 分支里 `vite.middlewares` 必须先运行。否则 `/@vite/client`、`/src/main.tsx`、CSS、HMR websocket 相关请求不会被 Vite 正确处理。

## 生产构建脚本

```json filename="package.json"
{
  "scripts": {
    "dev": "node server.js",
    "build": "vite build --outDir dist/client && vite build --ssr src/entry-server.tsx --outDir dist/server",
    "preview": "node server.js --prod"
  }
}
```

生产模式不使用 `ssrLoadModule`，因为生产环境不应依赖 Vite dev server。`dist/server/entry-server.js` 是已经构建好的服务端入口，可以直接 `import()`。

## 验证清单

1. 访问 `http://localhost:3000`。
2. 查看页面源码，`#root` 内应该有完整 HTML，而不是空容器。
3. 点击按钮，确认 hydration 后事件正常。
4. 修改 `src/App.tsx`，确认 dev 下页面更新。
5. 运行 `pnpm build && pnpm preview`，确认生产路径正常。
6. 故意在 `entry-server.tsx` 抛错，确认堆栈能指回源码。

## 什么时候该放弃手写 SSR

手写 SSR 适合学习底层机制，或做很薄的内部框架。生产项目通常还需要:

- 路由匹配
- 数据加载与缓存
- streaming
- head/meta 管理
- preload 链接生成
- 错误边界
- 安全序列化
- 部署平台适配

如果这些都要自己做，应该考虑 Remix、Nuxt、SvelteKit、React Router Framework Mode、Astro 或已有 Vite SSR 集成。理解本章的价值在于:你能看懂这些框架为什么这么封装。
