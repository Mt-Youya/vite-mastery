## `vite build` 仍是 Vite 命令，不是直接裸跑 Rolldown

Vite 8 的生产构建使用 Rolldown 打包，但 `vite build` 并不等于“把你的配置原样交给 Rolldown”。Vite 仍然负责:

1. 读取并合并 `vite.config.ts`
2. 加载 mode 与 `.env` 文件
3. 创建插件容器并安排 Vite 内置插件
4. 把 HTML、CSS、静态资源、worker、WASM 等 Vite 语义转成 Rolldown 能理解的输入
5. 调用 Rolldown 生成 chunks/assets
6. 输出 manifest、license、SSR manifest 等 Vite 产物

所以调构建配置时要分清两层:

| 层 | 配置入口 | 例子 |
| -- | -------- | ---- |
| Vite 构建语义 | `build.*` | `outDir`、`target`、`manifest`、`minify` |
| 底层 bundler 选项 | `build.rolldownOptions` | `input`、`external`、`output`、Rolldown 插件 |

优先使用 Vite 提供的上层配置；只有需要直接控制 bundler 行为时，才进入 `build.rolldownOptions`。

## `rolldownOptions` 是 Vite 8 主线

多入口、external、输出文件名等底层选项应写在 `build.rolldownOptions`:

```ts filename="vite.config.ts"
import { resolve } from "node:path"
import { defineConfig } from "vite"

export default defineConfig({
  build: {
    sourcemap: true,
    manifest: true,
    rolldownOptions: {
      input: {
        index: resolve("index.html"),
        admin: resolve("admin/index.html"),
      },
      external: ["some-cdn-lib"],
      output: {
        entryFileNames: "assets/[name]-[hash].js",
        chunkFileNames: "assets/[name]-[hash].js",
        assetFileNames: "assets/[name]-[hash][extname]",
      },
    },
  },
})
```

`build.rollupOptions` 在 Vite 8 中仍可读，但它只是 `build.rolldownOptions` 的 alias，并且官方配置参考已标记为 deprecated。新代码不要继续写旧名字。

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    // 旧配置仍能帮助迁移，但不建议新项目继续使用
    rollupOptions: {
      input: "index.html",
    },
  },
})
```

<Callout type="warning" title="迁移建议">
  迁移旧项目时，可以先保持 `rollupOptions` 让构建跑通；等验证完成后，把字段名改成 `rolldownOptions`，并逐项确认 output / plugin 选项是否仍是你需要的行为。
</Callout>

## 关键 build 配置速查

| 配置项 | 默认/说明 | 什么时候改 |
| ------ | --------- | ---------- |
| `build.outDir` | 默认 `"dist"` | 部署系统要求其他目录 |
| `build.assetsDir` | 默认 `"assets"` | 静态资源路径有约定 |
| `build.target` | 默认 `baseline-widely-available` | 需要更低浏览器目标 |
| `build.sourcemap` | 默认 `false` | 线上错误追踪、调试 |
| `build.manifest` | 默认 `false` | 后端模板要读产物映射 |
| `build.ssrManifest` | 默认 `false` | SSR 需要 preload / style 信息 |
| `build.minify` | client 默认 Oxc，SSR 默认 `false` | 调试、兼容 terser 选项 |
| `build.cssMinify` | 默认 Lightning CSS | CSS 压缩需要特殊目标 |
| `build.chunkImportMap` | 实验性，默认 `false` | 优化 chunk 缓存命中 |

这些配置里，`target`、`minify`、`cssMinify` 很容易被误用。先确认真实目标，再改默认值。

## target:浏览器能力边界

生产构建默认目标是 `baseline-widely-available`。这不是“支持所有旧浏览器”，而是以 Web Platform Baseline Widely Available 为参考，面向一组现代但相对稳妥的浏览器版本。

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    target: "es2020",
  },
})
```

降低 target 只处理语法转换，不等于自动补齐所有 polyfill。比如某个运行时 API 缺失，仍需要你自己引入 polyfill 或使用 legacy 方案。

## minify:Oxc 是主线，esbuild 是兼容项

Vite 8 client build 默认使用 Oxc minifier，SSR build 默认不压缩:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    minify: "oxc", // true 等价于默认策略；也可设 false / "terser" / "esbuild"
  },
})
```

`build.minify: "esbuild"` 已不应作为新配置首选。只有在迁移期间确认某个产物依赖 esbuild 压缩行为时，才临时使用。

调试生产产物时可以关压缩:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    minify: false,
    sourcemap: true,
  },
})
```

## CSS 压缩:Lightning CSS

CSS 压缩默认使用 Lightning CSS:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    cssMinify: "lightningcss",
  },
})
```

如果只想临时排查 CSS 压缩问题，可以单独关闭:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    cssMinify: false,
  },
})
```

不要为了排查 JS 压缩问题就同时关掉 CSS 压缩。Vite 允许把 JS 和 CSS minify 分开控制，应该分层定位。

## 多入口构建

Vite 默认使用 `<root>/index.html` 作为入口。多页面应用可以显式声明多个 HTML:

```ts filename="vite.config.ts"
import { resolve } from "node:path"
import { defineConfig } from "vite"

export default defineConfig({
  build: {
    rolldownOptions: {
      input: {
        index: resolve("index.html"),
        dashboard: resolve("dashboard.html"),
        settings: resolve("settings.html"),
      },
    },
  },
})
```

输出大致是:

```text
dist/
  index.html
  dashboard.html
  settings.html
  assets/
    index-HASH.js
    dashboard-HASH.js
    settings-HASH.js
    shared-HASH.js
```

多入口项目最容易出错的是 `base`、共享 chunk、后端模板路径。部署前必须用 `pnpm preview` 或真实静态服务器验证每个入口。

## 后端入口构建

如果 HTML 不归 Vite 管，而是由后端模板输出，应把 JS/TS 文件作为入口，并开启 manifest:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    manifest: true,
    rolldownOptions: {
      input: {
        app: "src/main.ts",
        admin: "src/admin.ts",
      },
    },
  },
})
```

这种模式下，如果需要 modulepreload polyfill，要在入口文件手动导入:

```ts filename="src/main.ts"
import "vite/modulepreload-polyfill"
```

完整后端集成见 [7.8 · 后端集成](/docs/07-build-pipeline/08-backend-integration)。

## chunk import map:缓存优化而不是拆包策略

`build.chunkImportMap` 是实验性优化，用 import map 减少 chunk hash 变动带来的级联缓存失效:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    chunkImportMap: true,
  },
})
```

它解决的是“chunk URL 引用导致上游 chunk 跟着变”的缓存问题，不是代码分割策略本身。拆包策略仍应从动态 import、路由边界、依赖体积和 Rolldown 输出配置入手。

## 迁移检查清单

1. 把新配置写到 `build.rolldownOptions`，旧 `rollupOptions` 只作为过渡。
2. 检查 `build.minify: "esbuild"`，没有明确原因就移除或改回默认。
3. 如果后端读产物，开启 `build.manifest` 并验证 manifest key。
4. 多入口项目逐个打开产物页面验证。
5. 部署到子路径时同步校准 `base`。
6. 使用 experimental 选项时，在文章或项目文档里明确标注。
