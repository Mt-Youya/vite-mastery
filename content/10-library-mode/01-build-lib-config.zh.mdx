## 库模式解决的不是“部署”，而是“被别人 import”

应用模式产物直接部署给浏览器:

```text
index.html -> assets/index-HASH.js -> assets/index-HASH.css
```

库模式产物发布到 npm，被其他项目作为依赖消费:

```ts
import { Button } from "@myorg/ui"
```

所以两者关注点不同:

| 维度 | 应用模式 | 库模式 |
| ---- | -------- | ------ |
| 入口 | HTML | JS/TS public API |
| 使用者 | 浏览器 | 另一个项目的 bundler/runtime |
| 文件名 | hash 优先 | 稳定路径优先 |
| 依赖 | 通常打进应用包 | 常把框架列为 peer |
| package.json | 不影响浏览器入口 | 决定消费者能 import 什么 |
| CSS | 自动注入页面 | 需要考虑如何让消费者引入 |

写库时最重要的问题不是“能不能 build”，而是“消费者用 Node、Vite、Next、Vitest、tsc 时能不能解析到正确文件和类型”。

## 最小配置

```ts filename="vite.config.ts"
import { resolve } from "node:path"
import { defineConfig } from "vite"

export default defineConfig({
  build: {
    lib: {
      entry: resolve("src/index.ts"),
      formats: ["es", "cjs"],
      fileName: (format) => `index.${format === "es" ? "js" : "cjs"}`,
    },
  },
})
```

`src/index.ts` 应该只导出公共 API:

```ts filename="src/index.ts"
export { Button } from "./components/Button"
export { createTheme } from "./theme"
export type { ButtonProps, ThemeOptions } from "./types"
```

不要让消费者 import 深层私有路径:

```ts
import { internalToken } from "@myorg/ui/dist/private/theme" // 不稳定
```

如果某个路径要公开，就把它写进 `exports`。

## `build.lib` 字段

### `entry`

单入口:

```ts
lib: {
  entry: "src/index.ts",
}
```

多入口:

```ts
lib: {
  entry: {
    index: "src/index.ts",
    react: "src/react.ts",
    theme: "src/theme.ts",
  },
}
```

多入口适合明确公开多个子路径:

```ts
import { Button } from "@myorg/ui/react"
import { tokens } from "@myorg/ui/theme"
```

### `formats`

```ts
lib: {
  formats: ["es", "cjs"],
}
```

选择建议:

| 格式 | 适合场景 |
| ---- | -------- |
| `es` | 现代 bundler、tree-shaking、默认首选 |
| `cjs` | 兼容仍使用 `require()` 的工具链 |
| `umd` | CDN `<script>` 全局变量 |
| `iife` | 极少数直接浏览器全局场景 |

现代 npm 包通常先考虑 `["es"]` 或 `["es", "cjs"]`。只有明确要支持 CDN 全局变量时，才输出 UMD/IIFE。

### `name`

`name` 只在 `umd` / `iife` 这类全局变量格式中有意义:

```ts
lib: {
  name: "MyLibrary",
  formats: ["es", "umd"],
}
```

如果只输出 `es` / `cjs`，`name` 不是核心字段。

### `fileName` 与 `cssFileName`

```ts
lib: {
  fileName: (format, entryName) => `${entryName}.${format === "es" ? "js" : "cjs"}`,
  cssFileName: "style",
}
```

`fileName` 决定 JS 产物路径，`cssFileName` 决定库导入 CSS 时生成的 CSS 文件名。对发布包来说，文件名稳定比 hash 更重要，因为 `exports` 会指向这些路径。

## external 与 peerDependencies

React 组件库不应该把 React 打进自己的库产物:

```ts filename="vite.config.ts"
export default defineConfig({
  build: {
    lib: {
      entry: "src/index.ts",
      formats: ["es", "cjs"],
    },
    rolldownOptions: {
      external: ["react", "react-dom", "react/jsx-runtime"],
    },
  },
})
```

同时在 `package.json` 声明:

```json filename="package.json"
{
  "peerDependencies": {
    "react": ">=18",
    "react-dom": ">=18"
  },
  "devDependencies": {
    "react": "^19.0.0",
    "react-dom": "^19.0.0"
  }
}
```

为什么要 external:

1. 避免消费者应用里出现两份 React。
2. 让消费者控制 React 版本。
3. 减小库包体积。
4. 避免 hooks、context、renderer 因多实例失效。

不是所有依赖都应该 external。小型纯函数依赖如果是实现细节，可以打进库里；框架、运行时、宿主应用必须单例的依赖，应该做 peer。

## `exports`:真正的公开 API

现代包解析优先看 `exports`:

```json filename="package.json"
{
  "name": "@myorg/ui",
  "type": "module",
  "files": ["dist"],
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./theme": {
      "types": "./dist/theme.d.ts",
      "import": "./dist/theme.js",
      "require": "./dist/theme.cjs"
    },
    "./style.css": "./dist/style.css"
  },
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts"
}
```

注意几点:

- `exports` 没列出的路径，消费者不能稳定 import。
- `types` 应该和每个公开入口对应。
- `main` / `module` 是旧工具兼容，不应替代 `exports`。
- CSS 如果要消费者手动引入，也应作为公开子路径暴露。

消费者用法:

```ts
import { Button } from "@myorg/ui"
import { tokens } from "@myorg/ui/theme"
import "@myorg/ui/style.css"
```

## CSS 怎么交付

组件库有三种常见策略:

| 策略 | 用法 | 适合 |
| ---- | ---- | ---- |
| 单独 CSS 文件 | `import "@myorg/ui/style.css"` | 传统组件库 |
| 每组件 CSS side effect | 组件入口内部 import CSS | 小型内部库 |
| CSS-in-JS / runtime styles | JS 内生成样式 | 主题高度动态 |

Vite 库模式会把库里 import 的 CSS 输出为 CSS 文件。你需要决定消费者如何拿到它。

如果希望消费者显式引入:

```json filename="package.json"
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js"
    },
    "./style.css": "./dist/style.css"
  },
  "sideEffects": ["**/*.css"]
}
```

`sideEffects` 很重要。否则某些 bundler 可能认为 CSS import 可被 tree-shaking 掉。

## 类型声明不由 Vite 自动完整解决

Vite 负责 JS/CSS/assets 打包，不负责把复杂 TS 类型声明完整打好。通常需要配合:

- `tsc --emitDeclarationOnly`
- `vue-tsc`
- `tsdown` / `rolldown-plugin-dts`
- API Extractor

本章只讲 `build.lib`，类型声明详见 [10.2 · 类型声明文件生成](/docs/10-library-mode/02-type-declarations)。

## 本地消费验证

不要只看 `dist/` 是否生成。至少做一次“消费者项目”验证:

```bash
pnpm build
pnpm pack
```

然后在临时 Vite app 中安装 tarball:

```bash
pnpm create vite@latest consumer --template react-ts
cd consumer
pnpm add ../myorg-ui-1.0.0.tgz
pnpm dev
pnpm build
```

验证:

1. `import { Button } from "@myorg/ui"` 有类型提示。
2. `import "@myorg/ui/style.css"` 能加载样式。
3. `pnpm build` 不出现重复 React 或解析失败。
4. Node/CJS 消费路径如果声明了，也能跑通。

## 完整 React 组件库示例

```ts filename="vite.config.ts"
import { resolve } from "node:path"
import react from "@vitejs/plugin-react"
import { defineConfig } from "vite"

export default defineConfig({
  plugins: [react()],
  build: {
    lib: {
      entry: {
        index: resolve("src/index.ts"),
        theme: resolve("src/theme.ts"),
      },
      formats: ["es", "cjs"],
      fileName: (format, entryName) => `${entryName}.${format === "es" ? "js" : "cjs"}`,
      cssFileName: "style",
    },
    rolldownOptions: {
      external: ["react", "react-dom", "react/jsx-runtime"],
      output: {
        assetFileNames: "assets/[name][extname]",
      },
    },
  },
})
```
