## The one-sentence summary

**A Vite plugin is a Rollup plugin plus a few Vite-only hooks.** This means most Rollup plugins work in Vite out of the box, but the reverse is not always true.

## Shared hooks (used in both dev and build)

These hooks are called during both the dev and build phases:

- `options`
- `buildStart`
- `resolveId`
- `load`
- `transform`
- `buildEnd`
- `generateBundle` (build only)
- `writeBundle` (build only)

## Vite-only hooks

| Hook                     | When it fires                   | Typical use case                      |
| ------------------------ | ------------------------------- | ------------------------------------- |
| `config`                 | Before user config is resolved  | Inject defaults, modify aliases       |
| `configResolved`         | After config is finalized       | Read the final config for later setup |
| `configureServer`        | When the dev server starts      | Inject middleware                     |
| `configurePreviewServer` | When the preview server starts  | Same as above, but for `vite preview` |
| `transformIndexHtml`     | When `index.html` is processed  | Inject scripts, meta tags, or data    |
| `handleHotUpdate`        | When an HMR update is triggered | Customize hot update behavior         |

## A minimal Vite plugin

```ts
// my-plugin.ts
import type { Plugin } from "vite"

export function myPlugin(): Plugin {
  return {
    name: "my-plugin",
    resolveId(id) {
      if (id === "virtual:hello") return id
    },
    load(id) {
      if (id === "virtual:hello") {
        return `export default "hello from virtual module"`
      }
    },
  }
}
```

This is a complete Vite plugin. **It only uses Rollup hooks**, so it would also work in a standalone Rollup project.

## Decision tree: which type should I write?

```text
Does your plugin use the dev server, index.html, or HMR?
  ├── Yes → Write a Vite plugin (use enforce / apply to control scope)
  └── No  → Write a Rollup plugin; Vite will handle compatibility automatically
```
