## 可视化 Bundle

```bash
pnpm add -D rollup-plugin-visualizer
```

```ts filename="vite.config.ts"
import { visualizer } from "rollup-plugin-visualizer"

export default defineConfig({
  plugins: [
    visualizer({
      open: true, // 构建后自动打开浏览器
      gzipSize: true, // 显示 gzip 后大小
      brotliSize: true, // 显示 brotli 后大小
      filename: "stats.html",
      template: "treemap", // "treemap" | "sunburst" | "network"
    }),
  ],
})
```

## 如何读懂可视化报告

**Treemap 视图**(推荐):方块面积代表文件大小

```text
┌─────────────────────────────────┐
│  react-dom (120KB)              │
│  ┌───────────────────────────┐  │
│  │ react-dom.development.js  │  │
│  └───────────────────────────┘  │
├─────────────────────────────────┤
│  antd (245KB) ← 可能太大了!     │
├─────────────────────────────────┤
│  lodash-es (8KB)                │
└─────────────────────────────────┘
```

**关注点**:

- 体积最大的方块:值得优化
- 意外出现的包:可能是幽灵依赖
- 重复的包:同一个包被打包了两次

## 常见优化策略

### 策略一:换更轻量的替代品

```text
lodash → lodash-es (按需 tree-shake)
moment → dayjs (2KB vs 300KB)
axios → ky 或 fetch (浏览器原生)
antd → 按需引入特定组件
```

### 策略二:动态 import 延迟加载

```ts
// 把图表库的初始加载延迟到用户真正需要时
const ChartComponent = lazy(() => import("./heavyChartComponent"))
```

### 策略三:检查是否引入了 Dev 依赖

```ts
// 确保开发工具不进入生产 bundle
import { debug } from "debug" // ← 是否在生产代码里用了这个?
```

## CI 里监控 bundle 大小

用 `bundlesize` 或 `size-limit` 在 CI 里设置体积预算:

```bash
pnpm add -D size-limit @size-limit/preset-app
```

```json filename="package.json"
{
  "size-limit": [
    {
      "path": "dist/assets/*.js",
      "limit": "300 KB" // 超过就 CI 失败
    }
  ]
}
```

或者用 GitHub Actions 的 `preinstall-size-comment` action 在 PR 里自动评论 bundle 大小变化。
