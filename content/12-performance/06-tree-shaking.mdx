## Tree-shaking 的工作原理

Tree-shaking(摇树优化)是 bundler 的能力:删除代码里**没有被用到的 export**。

```ts
// utils.ts
export function add(a: number, b: number) {
  return a + b
}
export function multiply(a: number, b: number) {
  return a * b
}

// main.ts
import { add } from "./utils"
console.log(add(1, 2))
// multiply 没有被 import → tree-shaking 删除它
```

**前提条件**:只有 ES Module(`import` / `export`)才能被 tree-shake。CommonJS(`require` / `module.exports`)无法 tree-shake。

## `sideEffects` 字段

```json filename="package.json"
{
  "name": "@my/utils",
  "sideEffects": false
}
```

`sideEffects: false` 告诉 bundler:

> 这个包里的所有模块都没有副作用——如果你没有 import 它的任何导出,可以完全安全地删除对这个文件的 import。

**什么是副作用**:

- 修改全局对象(`window.myLib = ...`)
- 注册全局事件监听器
- Polyfill
- CSS 导入

如果 CSS 文件有副作用:

```json
{
  "sideEffects": [
    "*.css",
    "./src/polyfills.ts" // 只有这两类有副作用
  ]
}
```

## 常见阻止 Tree-shaking 的模式

### 问题一:CommonJS 格式

```ts
// ❌ CommonJS:无法 tree-shake
const { debounce } = require("lodash")

// ✅ ESM:可以 tree-shake
import { debounce } from "lodash-es"
```

### 问题二:动态 property 访问

```ts
// ❌ 动态访问:bundler 不知道用了哪个方法
const key = "debounce"
const fn = lodash[key]

// ✅ 静态:可以分析
import { debounce } from "lodash-es"
```

### 问题三:有副作用的初始化代码

```ts
// ❌ 顶层代码有副作用
console.log("模块已加载!") // 即使没有 import 任何东西,这行也不会被删除

export function foo() {
  /* ... */
}
```

### 问题四:类的方法

```ts
// ❌ class 的方法不能单独 tree-shake
class MyUtils {
  add(a: number, b: number) {
    return a + b
  }
  multiply(a: number, b: number) {
    return a * b
  }
}

// ✅ 独立函数可以 tree-shake
export function add(a: number, b: number) {
  return a + b
}
export function multiply(a: number, b: number) {
  return a * b
}
```

## 验证 Tree-shaking 是否生效

方法一:检查产物文件

```bash
pnpm build
grep -r "multiply" dist/  # 如果有输出,说明没有被 tree-shake
```

方法二:用 bundle analyzer

用 `rollup-plugin-visualizer` 看看 `multiply` 函数是否出现在产物里。

方法三:用特定工具

```bash
pnpm add -D rollup-plugin-treeshaker  # 社区工具
```
