## 冷启动的时间组成

```text
pnpm dev 耗时 =
  Node.js 进程启动  (固定开销,~200ms)
  + 读取 vite.config.ts  (~50ms)
  + 依赖预构建  (首次:几秒~几十秒;缓存命中:<100ms)
  + 开启 HTTP server  (~50ms)
  + 等待第一个请求  (懒加载,不计入)
```

**最大变量**:依赖预构建的时间。

## 为什么预构建会很慢

1. **依赖太多**:项目用了大量 npm 包
2. **CJS 包需要转换**:CommonJS → ESM 需要扫描代码
3. **嵌套依赖**:深层依赖也需要预构建
4. **首次没有缓存**:`.vite/deps/` 不存在

## 加速技巧一:预声明所有依赖

Vite 默认通过扫描源码自动检测依赖。但如果代码里有动态 import,Vite 可能遗漏一些包——导致启动后第一次请求时才预构建,触发页面刷新。

主动声明让 Vite 一次预构建完所有:

```ts filename="vite.config.ts"
export default defineConfig({
  optimizeDeps: {
    include: [
      // 显式列出所有重要依赖(包括动态 import 的)
      "react",
      "react-dom",
      "react-router-dom",
      "axios",
      "lodash-es",
      // 某些包的子路径
      "lodash-es/debounce",
      "lodash-es/throttle",
    ],
  },
})
```

## 加速技巧二:排除不需要预构建的包

```ts filename="vite.config.ts"
export default defineConfig({
  optimizeDeps: {
    exclude: [
      // 已经是 ESM 且单文件的包不需要预构建
      "@vite-mastery/ui", // workspace 本地包
      "some-pure-esm-package", // 现代 ESM 包
    ],
  },
})
```

## 加速技巧三:monorepo 里跳过本地包

```ts filename="vite.config.ts"
export default defineConfig({
  optimizeDeps: {
    exclude: [
      // 所有 workspace 包不预构建
      "@myorg/ui",
      "@myorg/utils",
      "@myorg/hooks",
    ],
  },
  resolve: {
    // 直接 alias 到源码,省去 resolve 步骤
    alias: {
      "@myorg/ui": resolve("../../packages/ui/src"),
    },
  },
})
```

## Vite 8 Rolldown 的改善

Vite 8 用 Rolldown 做依赖预构建,相比 Vite 7 的 esbuild:

- **更快的 CommonJS 转换**:OXC Parser 比 esbuild 在某些场景更快
- **更好的缓存利用率**:Rolldown 的 transform 管道与 build 共用,减少重复工作
- **增量预构建**:只重建真正变化的依赖

实际感受:大型项目首次冷启动可能快 20~50%。

## 调试慢启动

```bash
# 查看 Vite 的内部日志,找慢在哪
DEBUG=vite:deps pnpm dev

# 或者用 --debug 标志
pnpm vite --debug deps
```

输出示例:

```text
vite:deps Optimizing dependencies...
vite:deps  react     +312ms
vite:deps  react-dom +1234ms  ← 最慢的包
vite:deps  lodash-es +89ms
```

## 不要盲目 include 全世界

`optimizeDeps.include` 是用来补充 Vite 扫描不到的依赖,不是越多越好。过度 include 会带来两个问题:

1. 首次预构建变慢,因为 Vite 需要处理更多包。
2. 缓存更容易失效,锁文件或依赖版本一变就要重建更多内容。

推荐流程:

1. 先不写 `include`,用默认扫描跑一次。
2. 打开 `DEBUG=vite:deps`,观察哪些依赖在页面请求后才触发二次预构建。
3. 只把这些漏掉的依赖加入 `include`。
4. 记录原因,例如“由动态 import 字符串拼接引入,扫描器无法静态发现”。

这样配置是可解释的,而不是一份没人敢删的依赖名单。

## 冷启动优化清单

| 检查项 | 为什么影响冷启动 |
| --- | --- |
| 锁文件是否频繁变化 | 依赖预构建缓存会跟着失效 |
| 是否在 `vite.config.ts` 里做同步 I/O | 配置加载阻塞 dev server 启动 |
| 插件是否在 `configResolved` 扫全仓库 | dev 启动前就做了大量工作 |
| workspace 包是否被误预构建 | 本地源码包通常应该直接由 Vite 转换 |
| 是否把大型可选依赖放进首屏 | 首次页面请求会拉长依赖处理链 |

优化时先测量,再改配置。冷启动慢可能来自依赖预构建,也可能来自配置文件、插件初始化或网络代理。把所有问题都归咎于 `optimizeDeps` 会误导排查方向。
