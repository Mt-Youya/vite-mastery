## 原生 ESM dev 的局限

Vite 的 dev server 默认使用原生 ESM:每个模块单独请求。这带来了极快的冷启动,但也有代价:

**问题一:HTTP 请求数量多**

大型项目可能有几千个模块。浏览器并发请求数有限(HTTP/2 也不是无限的),首次加载时可能出现明显延迟。

**问题二:Dev/Build 行为差异**

即使 Rolldown 统一了 transform,dev 模式仍然没有 bundle——没有 tree-shaking、没有 chunk 分割。某些只在 bundle 后才能发现的问题在 dev 阶段看不出来。

**问题三:无法模拟 CDN 场景**

某些第三方库的 CDN 版本行为与 npm 版本不同,原生 ESM dev 无法模拟这种场景。

## Full Bundle Mode 的思路

Full Bundle Mode 让 dev server 也做完整 bundle:

```text
当前 dev 模式:
  浏览器请求 /src/App.tsx
    → Vite transform (单文件)
    → 返回单个 ESM 模块
    → 浏览器再请求依赖...
  (几十到几千个请求)

Full Bundle Mode:
  Vite 用 Rolldown 构建一个 dev bundle
    → 所有模块打包(但不压缩)
    → 仍然支持 HMR(增量 bundle)
    → 只有 1~几个请求
```

## 权衡

| 维度             | 当前 ESM dev       | Full Bundle Mode  |
| ---------------- | ------------------ | ----------------- |
| 冷启动速度       | 极快(不 bundle)    | 较慢(需要 bundle) |
| 每次 HMR         | 只更新改变的模块   | 增量 rebundle     |
| 请求数量         | 很多(每个模块一个) | 少(几个 chunk)    |
| Dev/Build 一致性 | 有差异             | 更接近 build      |

**适合场景**:网络延迟高的环境、需要精确模拟生产行为的场景。
