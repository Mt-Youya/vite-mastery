## 功能回顾

`unplugin-auto-import` 让你可以这样写代码:

```ts
// 不需要 import!
const count = ref(0)
const doubled = computed(() => count.value * 2)
```

插件自动在文件顶部注入:

```ts
import { ref, computed } from "vue"
```

## unplugin:跨 bundler 插件框架

`auto-import` 使用 `unplugin` 框架,让一套代码能在 Vite / Webpack / Rollup / esbuild 上运行:

```ts
// 源码结构(简化)
import { createUnplugin } from "unplugin"

const AutoImport = createUnplugin((options) => {
  return {
    name: "unplugin-auto-import",
    transform(code, id) {
      /* ... */
    },
    // ...
  }
})

// 导出各 bundler 的版本
export const vitePlugin = AutoImport.vite
export const webpackPlugin = AutoImport.webpack
export const rollupPlugin = AutoImport.rollup
```

`unplugin` 在内部把这个统一的 hook 格式适配到各 bundler 的 API。

## 核心逻辑:静态 import 检测

```ts
// 简化的核心 transform 实现
function transform(code: string, id: string): string | null {
  // 1. 跳过不需要处理的文件
  if (!filterFile(id)) return null

  // 2. 解析代码,找出哪些标识符没有 import
  const usedIdentifiers = scanUsedIdentifiers(code)
  const existingImports = parseExistingImports(code)

  // 3. 找出需要自动注入的 import
  const toInject: Record<string, string[]> = {}
  for (const identifier of usedIdentifiers) {
    if (existingImports.has(identifier)) continue // 已经 import 了
    const source = lookupSource(identifier, options.imports)
    if (!source) continue // 不在配置的 preset 里

    toInject[source] ??= []
    toInject[source].push(identifier)
  }

  if (Object.keys(toInject).length === 0) return null

  // 4. 生成 import 语句
  const importStatements = Object.entries(toInject)
    .map(([source, names]) => `import { ${names.join(", ")} } from "${source}"`)
    .join("\n")

  return `${importStatements}\n${code}`
}
```

## 标识符扫描的实现

标识符扫描是关键挑战:如何知道 `ref` 是用到了,还是只是一个变量名?

简单实现(正则):

```ts
function scanUsedIdentifiers(code: string): Set<string> {
  const identifiers = new Set<string>()
  const IDENTIFIER_RE = /\b([a-zA-Z_$][a-zA-Z0-9_$]*)\b/g

  let match: RegExpExecArray | null
  while ((match = IDENTIFIER_RE.exec(code)) !== null) {
    identifiers.add(match[1])
  }

  return identifiers
}
```

更准确的实现(AST):unplugin-auto-import 实际使用 AST 来精确检测标识符使用,而不是简单的正则。

## .d.ts 自动生成

插件在 `buildEnd` 里生成全局类型声明:

```ts
buildEnd() {
  if (!options.dts) return

  const declarations = generateDtsContent(resolvedImports)
  writeFileSync(resolve(options.dts), declarations)
},

function generateDtsContent(imports: ImportMap): string {
  const lines = ["// 自动生成,勿手动修改"]
  lines.push("export {}")
  lines.push("declare global {")

  for (const [identifier, source] of imports) {
    lines.push(
      `  const ${identifier}: typeof import("${source}")["${identifier}"]`
    )
  }

  lines.push("}")
  return lines.join("\n")
}
```

## 源码阅读顺序

读 `unplugin-auto-import` 不建议从 transform 函数直接跳进去。更稳的顺序是:

1. **入口导出**:先看它如何通过 `createUnplugin` 导出 Vite、Rollup、Webpack 等不同适配版本。
2. **options 解析**:看 presets、dirs、imports、dts、eslintrc、include/exclude 如何被归一化。
3. **扫描器**:看插件如何从源码里找未声明标识符,以及如何避免把属性名、局部变量、类型名误判成全局 API。
4. **注入器**:看 import 语句如何插到正确位置,如何保留 shebang、注释、source map。
5. **声明文件生成**:看 `.d.ts` 和 ESLint globals 如何根据最终 import map 生成。
6. **缓存与 watcher**:看目录扫描、配置变化、类型文件更新如何避免每次 transform 都全量重算。

源码里真正重要的不是“加一行 import”,而是围绕这行 import 建立的一整套索引、过滤、缓存和类型体验。

## 调用流程

一次文件 transform 大致可以拆成这条流程:

```text
用户保存文件
  -> Vite 触发 transform(code, id)
  -> include/exclude 判断是否处理
  -> AST/词法扫描收集候选标识符
  -> 排除已有 import、局部声明、类型位置
  -> 根据 import map 找到来源包
  -> 用 MagicString 注入 import
  -> 返回 code + sourcemap
  -> 更新用到的标识符缓存
  -> buildEnd / watcher 阶段刷新 d.ts
```

这条流程解释了为什么 auto-import 不能只靠 `resolveId`。`resolveId` 只能决定一个模块路径如何解析,但 auto-import 的问题是“当前源码里缺了哪些 import”。这个判断必须看到完整源码,所以核心逻辑自然落在 `transform`。

## 边界与坑点

<Callout type="warning" title="auto-import 插件最容易踩的坑">
  - 正则扫描会把对象属性、字符串、注释里的名字误认为变量
  - 没有排除已有 import 会生成重复导入
  - 没有区分 type-only 使用会把类型引用变成运行时 import
  - `.d.ts` 更新不及时会导致编辑器提示和实际 transform 不一致
  - barrel preset 太大时,每个文件都扫描全量 API 会拖慢 HMR
</Callout>

生产级实现通常会用 AST、作用域分析和 MagicString。AST 解决“这个标识符到底是不是自由变量”,作用域分析解决“它是不是已经在本文件声明”,MagicString 解决“插入 import 后 sourcemap 还能不能用”。
