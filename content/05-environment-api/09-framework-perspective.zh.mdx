## 框架面临的问题

在 Vite 7 及之前,主流 meta 框架都要独立解决同一个问题:

> **如何在一个 Vite dev server 里同时运行 client 代码和 server 代码,同时让 HMR 在两端都工作?**

每个框架的解法各不相同,很多是"私有 API hack":

- 直接操作 Vite 内部状态
- 多次实例化 Vite server
- 自己维护模块图谱

Environment API 的出现提供了**标准化的接口**,让框架不再需要 hack。

## Nuxt 的适配思路

Nuxt 3 的 SSR 架构需要:

- `client` 环境:Vue 应用的浏览器端代码
- `server` 环境:Nitro 服务端(不是普通 Node.js SSR,是 Nitro 的 H3 应用)
- 可选的 `edge` 环境:部署到 CloudFlare/Vercel Edge 时

Nuxt 的 Environment API 适配大致方向:

```ts
// nuxt-vite-plugin(概念示意)
export function nuxtVitePlugin(): Plugin {
  return {
    name: "nuxt:environment",
    config() {
      return {
        environments: {
          client: {
            // Vue 浏览器端配置
          },
          server: {
            // Nitro 服务端配置
            resolve: {
              conditions: ["node", "import"],
            },
          },
        },
      }
    },
    async buildApp(app) {
      // 先构建 server(Nitro 需要 Vue 组件的 SSR manifest)
      await app.build("server")
      await app.build("client")
    },
  }
}
```

**应用开发者感受不到变化**——你仍然只需要写 Vue 组件和 Nuxt 配置。Env API 的复杂度被 Nuxt 内部吸收了。

## SvelteKit 的适配思路

SvelteKit 的特点是有非常精细的 server/client 代码划分:

- `.server.ts` 文件只在服务端加载
- `+page.server.ts` 只在 server 环境执行
- `+page.ts` 在 client 和 server 都执行(同构)

这天然对应 Environment API 的多环境模型:

```ts
// sveltekit-vite-plugin(概念示意)
export function sveltekitPlugin(): Plugin {
  return {
    name: "sveltekit:environment",
    config() {
      return {
        environments: {
          client: {
            resolve: {
              conditions: ["browser", "svelte", "import"],
            },
          },
          ssr: {
            resolve: {
              conditions: ["node", "svelte", "import"],
              noExternal: ["svelte"],
            },
          },
        },
      }
    },
    transform(code, id) {
      // .server.ts 文件只允许在 ssr 环境 import
      if (id.endsWith(".server.ts") && this.environment?.name === "client") {
        return { code: 'throw new Error("server-only module")', map: null }
      }
      return null
    },
  }
}
```

## React Router v7 / Remix 的适配思路

React Router v7(Remix 的演进)需要支持 RSC(React Server Components):

```ts
// react-router-vite-plugin(概念示意)
export function reactRouterPlugin(): Plugin {
  return {
    name: "react-router:environment",
    config() {
      return {
        environments: {
          client: {
            resolve: {
              conditions: ["browser", "react-client", "import"],
            },
          },
          ssr: {
            resolve: {
              conditions: ["node", "import"],
            },
          },
          rsc: {
            resolve: {
              conditions: ["react-server", "node", "import"],
            },
          },
        },
      }
    },
    async buildApp(app) {
      // RSC → SSR → Client 的顺序
      await app.build("rsc")
      await app.build("ssr")
      await app.build("client")
    },
  }
}
```

## 框架对应用开发者的屏蔽层

所有框架都有一个共同点:对应用开发者**屏蔽 Env API 的复杂度**:

```text
应用开发者看到的:
  写 .vue 文件  /  写 .svelte 文件  /  写 .tsx 文件
       │
框架层屏蔽:
  Vite Plugin + Environment API 配置
       │
底层:
  client Environment / ssr Environment / rsc Environment
```

作为应用开发者,你通常不需要直接使用 Environment API。你只需要:

- 使用框架约定(`.server.ts` 后缀、server-only 导入等)
- 配置框架的选项(不是 Vite 的 `environments` 选项)

## 何时需要直接用 Env API

作为应用开发者,以下情况才需要直接接触 Environment API:

1. **自定义部署环境**:公司内部有特定的 runtime(非标准 Node.js)
2. **写 meta 框架或框架插件**:你在帮助框架实现 Env API 支持
3. **多 Worker 架构**:项目本身是 monorepo,不同 Worker 需要不同 Environment
