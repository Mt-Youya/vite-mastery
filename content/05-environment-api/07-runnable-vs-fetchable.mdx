## 两种通信模型

`DevEnvironment` 的两个子类对应了两种不同的模块执行模型:

### RunnableDevEnvironment

模块在 **Vite dev server 的同一个 Node.js 进程**里执行:

```text
[Node.js 进程]
  ├── Vite Dev Server
  │     └── DevEnvironment (ssr)
  │           └── ModuleRunner
  │                 └── 直接 eval / Function 执行模块代码
  │
  └── 你的 SSR 服务器代码
        └── runner.import("/src/entry.ts")  ← 同进程调用
```

**优点**:

- 零通信开销
- 共享内存(全局变量、模块状态)
- 调试方便(同一个 Node.js inspector)

**缺点**:

- 只能在 Node.js 里运行(无法模拟 Edge Runtime)
- 模块之间没有真正的隔离

**适合**:标准 Node.js SSR、传统服务端渲染。

### FetchableDevEnvironment

模块在**独立的进程或 runtime**里执行,Vite server 通过 HTTP 或 IPC 提供模块:

```text
[Node.js 进程]              [独立进程/runtime]
  Vite Dev Server     ←→    你的代码运行在这里
  DevEnvironment (worker)   例如:
  提供 transform 服务         - workerd (Cloudflare)
  接受 module fetch 请求       - Deno
  提供 HMR 通知                - 另一个 Node.js 进程
```

**优点**:

- 真正的 runtime 隔离(在 workerd 里测试 Cloudflare 代码)
- 更真实地模拟生产环境

**缺点**:

- 通信开销(HTTP/IPC)
- 调试更复杂
- 配置更复杂

**适合**:Cloudflare Workers 开发、Edge Runtime 开发、隔离需求高的场景。

## RunnableDevEnvironment 的基本用法

大多数 SSR 场景使用 RunnableDevEnvironment,这也是 Vite 的默认行为:

```ts
// 标准 SSR 配置,默认使用 RunnableDevEnvironment
export default defineConfig({
  environments: {
    ssr: {
      resolve: {
        conditions: ["node", "import"],
      },
    },
  },
})

// server.js 中:
const runner = getModuleRunner(vite.environments.ssr)
const { render } = await runner.import("/src/entry-server.tsx")
// render() 在同一个 Node.js 进程里执行
```

## FetchableDevEnvironment 的原理

对于 FetchableDevEnvironment,模块执行发生在**另一个 runtime**里:

```text
1. 你在 workerd 里启动一个 worker
2. worker 里的 import 请求发给 Vite dev server
3. Vite server 执行 resolve + transform
4. 返回编译后的模块代码给 workerd
5. workerd 执行这段代码
6. 文件改变时,Vite 通过 HotChannel 通知 workerd 刷新
```

这意味着 FetchableDevEnvironment 需要在目标 runtime 里有一个**配套的 client 端**来接收模块。

```ts filename="vite.config.ts"
// 伪代码:Cloudflare Workers FetchableDevEnvironment 配置

      // 具体 API 以官方文档和 Cloudflare Vite plugin 为准

  },

```

实际上,对于 Cloudflare Workers 开发,更推荐使用:

- `@cloudflare/vite-plugin` — Cloudflare 官方 Vite 插件
- `wrangler dev` — Cloudflare CLI 工具

这些工具内部实现了 FetchableDevEnvironment 所需的所有胶水代码。

## 如何选择

```text
你的代码需要在 Node.js 环境里运行?
  ├── 是 → 用 RunnableDevEnvironment(默认)
  └── 否 → 你需要模拟特定 runtime:
        ├── Cloudflare Workers → 用 Cloudflare 官方 Vite plugin
        ├── Deno Deploy → 等待 Deno 官方集成
        └── 自定义 Edge → 实现 FetchableDevEnvironment(复杂)
```
